## Lessons learned in Session 6 (apply or watch for in Session 7)

These patterns surfaced in Session 6's eleven rounds of fresh reviewers and are worth carrying into Session 7 from the start rather than learning them via review iterations:

1. **The trajectory is bouncing, not strictly monotonic.** Session 6 trajectory: R1=25, R2=10, R3=19, R4=3, R5=5, R6=14, R7=5, R8=4, R9=8, R10=4, R11=6. Total: 93 distinct findings applied across 11 rounds. The bouncing reflects that each round-N fix can introduce new consistency gaps elsewhere that round-(N+1) reviewers catch. **Plan for 4–11 rounds; commit at the user's stopping criterion rather than chasing zero findings.** Session 6 committed at post-11th-pass state per user's explicit stop signal.

2. **Fixes can introduce defects.** Multiple findings in later rounds were defects I introduced when fixing earlier-round findings. Examples: round-8 fix for R8B's structural finding (added "On section placement" paragraph) introduced a plant-watering violation that R11C caught; round-10 fix for R10A's missing-step-name finding introduced a card-note/activity-log conflation that R11A caught. **Watch every fix for whether it introduces a new defect; reviewer rounds will surface these but it costs cycles.**

3. **Description-to-process consistency must be maintained.** When the Failure modes section says X, the Process step text must say X too. Multiple findings caught defects where I updated one but not the other. **For every change to a Failure modes entry, check whether the corresponding Process step needs the same change. For every change to a Process step's halt instruction, check whether the corresponding Failure modes entry needs the same change.**

4. **Halt-condition completeness is four-component plus "Stop."** Every halt path needs: (a) trigger condition, (b) card note content with specific MCP call, (c) activity log content with specific MCP call and step name, (d) post-halt disk/artifact state cleanup attestation, (e) terminal "Stop." The "Stop." was easy to miss; R10A and R11C both caught instances where I had four components but no "Stop."

5. **Plant-watering on explanatory sentences.** Round 8 added a placement-rationale paragraph that contained third-person declarative explanations ("Operational correctness is preserved because..."). Round 9 caught the commit hashes and plan references. Round 11 caught the remaining "Operational correctness is preserved" sentence. **Every sentence added to a profile must be an instruction to the subagent — no third-person declarative explanations, no commit hashes, no plan §N pointers, no audit-trail annotations.**

6. **Cross-call-site consistency for repeated requirements.** The "Source decisions L1 form non-compliance" rule needed to be named at five call sites (Inline disciplines table, Output contract, Step 6(b), Step 7 Card Slices section requirements, Step 8 gate check (b)). Multiple rounds caught defects where I'd updated some but not all. **For any rule that appears at multiple call sites, the reviewer brief should explicitly enumerate all call sites and the reviewer should mechanically check each one.**

7. **Mechanical asymmetries are caught.** Round 6 surfaced that I'd moved the `rules_fired` absent-key check inline to its bullet but left the `blast_radius` absent-key check at the end of Step 2 — asymmetric handling. Round 11 surfaced that the non-empty-Glob and empty-Glob-other-reason Write failures explicitly included "file path in card note" but the empty-Glob-parent-directory-missing case did not. **For every pair/triple/quintuple of parallel cases, verify the parallel handling is symmetric across all instances.**

8. **R-DOC-2 italic attestation line is a required document element, not a heading.** The validation hook's R-DOC-2 L1 sequence is: `# Architecture —`, `## Goal`, `## Scope`, italic-attestation line, `## Card Slices`, `## Limitations`, `## Standards governing this architecture`, `## Status`. The italic line is between Scope and Card Slices and is explicitly required.

9. **R-REVIEW-1/2/3 (validation hook) for `ARCH_DESIGN_REVIEW_V1`** — the design reviewer must produce JSON that satisfies the hook rule set. Plan §7: findings list with severity / category / summary / document_citation; `summary.{blocker_count, serious_count, minor_count}` matching actual counts.

10. **Multiple reviewers per round is load-bearing.** Sessions 4, 5, and 6 each used three parallel reviewers per round. Single-reviewer rounds would miss the cross-reviewer convergence signal that distinguishes "real defect" from "single reviewer opinion." A finding caught by ≥2 of 3 reviewers in the same round is high-confidence; single-reviewer findings still apply per the no-grading discipline but warrant a closer look.

## Design decisions locked in Sessions 2–6 — Session 7 inherits these

1. **No codebase discovery in compose.** (Compose-only constraint — does not apply to the design reviewer directly since it doesn't compose. But the design reviewer also has no `rag_search` / `codegraph_*` / `rag_query_impact` in its tools list per plan §6.6 frontmatter — it reasons from the verified bundle and the architecture document.)

2. **Compose retains Context7 (both `resolve-library-id` AND `query-docs`).** (Compose-only — the design reviewer doesn't need Context7 since it reads the architecture document and judges whether the document's claims are internally consistent, not whether external library docs match.)

3. **Universal expert-standards activation as Step 1, verbatim.** Non-negotiable. The design reviewer's Step 1 is the same activation text.

4. **Freestanding `## Halt condition` section.** Pattern from Sessions 4, 5, and 6. The design reviewer should follow the same pattern: section appears between Output contract and Process; Preamble's "Execution order" specifies execution after Step 1 first; Halt condition section has a self-enforcement reminder for premature execution. The Halt condition's specific checks for the design reviewer are TBD per plan §6.6 Step 1 — likely checks document parseability and required structural markers.

5. **Bundle alias binding.** `bundle` is bound in the preamble as an alias for the verified bundle the orchestrator passes. (For the design reviewer, this is `verified_bundle` fetched via `get_workspace_artifact` rather than passed inline — different mechanism, same alias discipline.)

6. **Plant-watering rule.** Every sentence in the profile is an instruction to the subagent the profile defines. The design reviewer's profile must satisfy this.

7. **R-DOC-2 is a subsequence check** (heading-order subsequence). The design reviewer reads the architecture document and may need to verify R-DOC-2 conformance as part of its review.

8. **Level representation split.** Machine contracts use numeric 1/2/3. Document marker uses `**Level:** L1`/`L2`/`L3`. The design reviewer reads `**Level:** L#` from the document.

9. **Reviewer discipline: review the work, not the fixes.** Every follow-up reviewer dispatch is briefed as an independent fresh review against contract / plan / standards, with no mention of prior findings. Saved as `feedback_review_not_verify.md` in local memory. Applied for compose-l1 in Session 6 across 11 rounds.

10. **All findings apply in full — no grading.** Never label issues as "worth flagging" / "near-miss" / "minor — won't fix." Saved as `feedback_no_grading_findings.md` in local memory. Applied in Session 6 — 93 findings across 11 rounds, all applied.

11. **Plan amendments are in scope for the producing session.** When a reviewer finding surfaces a plan-side defect caused by the session's profile work, fix the plan in the same commit. Session 6 considered amending plan §6.5 to enumerate the actual section structure (which exceeds the plan's terse 4-item summary) but chose not to amend, following Session 5 precedent for plan §6.4. Session 7 should consider whether plan §6.6's body structure summary needs amending after the design-reviewer profile is authored.

12. **Halt-condition four-component completeness plus "Stop."** Every halt path specifies: (a) trigger, (b) card note content, (c) activity log content, (d) post-halt state, (e) terminal "Stop." The design reviewer's halt paths should follow the same pattern.

13. **Failure modes section as standalone reference.** Every halt path in the body has a corresponding Failure modes entry; the Failure modes section is the consolidated reference a future maintainer reads. Both must be consistent on trigger, card note content, activity log content, and post-halt state.

14. **Description-to-process consistency.** When the Failure modes section says X, the Process step text must say X too. When the Process step's halt instruction names "step (Step N)" in the activity log, the Failure modes entry must also name "step (Step N)" in the activity log.

## Design decisions locked in Sessions 2–5 — Session 6 inherits these

1. **No codebase discovery in compose.** Frontmatter excludes every RAG and codegraph tool. Discovery results come from the bundle's design fields. This is the load-bearing rework change.

2. **Compose retains Context7 (`resolve-library-id` AND `query-docs`).** External doc lookup is reasoning-shaped and stays in compose. (L1 may have less use for Context7 in practice since L1 cannot introduce external libraries, but the tools remain available; mid-process discovery of an external library is a classification escalation halt.)

3. **Universal expert-standards activation as Step 1, verbatim.** Non-negotiable.

4. **Freestanding `## Halt condition` section.** Pattern from Sessions 4 and 5: section appears between Output contract and Process in document order; the preamble's "Execution order" list specifies execution as Step 1 first (cross-cutting), then Halt condition (parses only the level field), then Step 2 onward (full design-field ingestion). Halt checks both `verified_level` (orchestrator parameter) AND `bundle.rule_evaluation.computed_level` (bundle field), halts on disagreement with both values surfaced for orchestrator reconciliation.

5. **Bundle alias.** `bundle` is bound in the preamble as an alias for `arch_facts_bundle`; all `bundle.<field>` references are paths into that object. Single binding statement in the preamble, used consistently throughout.

6. **Corrected-bundle semantics.** When the auditor found discrepancies, the orchestrator passes the corrected bundle, and `bundle.rule_evaluation.computed_level` is the auditor's recomputed level (substituted in by the auditor's corrected-bundle construction per `agents/architecture-classification-auditor.md`). When no discrepancies, it is the research agent's original level. In either case it equals `verified_level`. Plan §3 was annotated in Session 4 to make this explicit.

7. **Level representation split.** Machine contracts use numeric 1/2/3. Document marker uses `**Level:** L1`/`L2`/`L3` as a standalone line matching the validation hook's R-DOC-1 pattern `^\*\*Level:\*\* L[123]$` exactly — no leading or trailing content on the same line. Compose translates numeric → marker form.

8. **Element 5 premise sources for V2.** Spec citation, referenced-document citation, bundle-field citation (specific entry like `bundle.design_fields.existing_patterns_hits[3]` with file/line/snippet), Context7 (library, version/date, what was confirmed), Read of a file the bundle's `files_relevant` named, Grep against spec or named document, test reproduction, or explicit "no factual premises — pure design choice". **Do not cite a file the bundle did not name in `files_relevant`, even when the bundle mentions the file in another design field** (e.g., `bundle.design_fields.blast_radius.for_candidate_modified_set[i].top_affected`). Codebase-RAG and CodeGraph removed entirely. **L1 note:** at L1 there are no D# decisions and therefore no element 5 — slices trace directly to R#/Q#; the element 5 discipline carries forward only for compose profiles that produce D# decisions (L2 and L3).

9. **R-DOC-2 is a subsequence check.** Plan §7 was clarified in Session 4: the required headings must appear in the specified relative order, but optional sections (`## Inheritance from existing precedents` at any level; `## Threat model` / `## ASVS verification mapping` at L3) may interleave without failing the rule. **L1 note:** R-DOC-2 L1 sequence per plan §7: `# Architecture —`, `## Goal`, `## Scope`, italic-attestation line, `## Card Slices`, `## Limitations`, `## Standards governing this architecture`, `## Status`.

10. **Two-condition family criterion for inheritance.** `## Inheritance from existing precedents` section is required only when both (a) the architecture addresses a structurally identical problem within the same system as a prior architecture AND (b) it uses the same architectural pattern. Step 3 in compose evaluates the criterion. (Rare at L1 — most L1 work is too small for an architecture family to exist around it.)

11. **Reviewer discipline: review the work, not the fixes.** Every follow-up reviewer dispatch must be briefed as an independent fresh review of the current file against contract / plan / standards, with no mention of prior findings in the brief. Verification-style briefing (listing prior findings and asking "was each fix applied correctly?") anchors the reviewer and is forbidden. Stop dispatching new passes when a fresh pass returns zero findings — not when "all prior findings verified applied." This pattern was load-bearing in Sessions 4 and 5: fresh passes surface 3–25 new defects each round, in different sections of the file. Convergence trajectory was diminishing but Session 5 committed at post-5th-pass state per user choice. Saved as `feedback_review_not_verify.md` in local memory.

12. **All findings apply in full — no grading.** Never label issues as "worth flagging" / "near-miss" / "minor — won't fix." If a reviewer notices it, apply it. This applies recursively to the author's self-review too. Saved as `feedback_no_grading_findings.md` in local memory.

13. **Plant-watering rule.** Every sentence in the profile is an instruction to the subagent the profile defines. No content directed at the orchestrator, user, plan author, or maintainer. Includes meta-content about what other levels (L2, L3) do — the L1 subagent does not need to know what L2 or L3 do; it needs to know what L1 does.

14. **Plan amendments are in scope for the producing session.** When a reviewer finding surfaces a plan-side defect caused by the session's profile work, fix the plan in the same commit. Pattern: Session 3 amended plan §3; Session 4 amended plan §3, §6.3, §7. Session 5 did not require a plan amendment but left a documented note that plan §6.4's body structure list is non-exhaustive (only 4 items while the actual profile structure has 9 sections per the L3 precedent) — Session 6 should consider whether to amend plan §6.5 for the same alignment.

15. **Two-pass write at L2/L3, single-pass at L1 (contract §"What stays" item 5).** L2 (Session 5) and L3 (Session 4) write the body in pass 1 and slice in pass 2. Gate-completion Edits at Step 10 (Status placeholder replacement, Gate A attestation incorporation) are not additional structural passes — they are gate-fixing edits that happen against the committed document. **L1 is single-pass** — the architecture is the slicing, so the document is written in one pass.

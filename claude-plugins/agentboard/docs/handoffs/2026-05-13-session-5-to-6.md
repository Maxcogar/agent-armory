## Lessons learned in Session 5 (apply or watch for in Session 6)

These patterns surfaced in Session 5's five rounds of fresh reviewers and are worth carrying into Session 6 from the start rather than learning them via review iterations:

1. **Description must match the contract's locked terminology.** Round 5's only Critical finding was that the frontmatter description said "three-pass write" while the contract §"What stays" item 5 locks "two-pass write at L2/L3." The fix was to reframe gate-completion Edits as not-additional-passes. **For L1, the contract locks "single-pass write" — the description must say single-pass, not multi-pass, and gate-completion edits (if any — and at L1 with a single delivery gate against a single-pass document, there may be none) must not be framed as additional passes.**

2. **Bundle-integrity cross-check at Step 2 is worth replicating.** Session 5's compose-l2 added: (a) halt if `bundle.design_fields.blast_radius` is absent entirely; (b) halt if `files_relevant` contains `candidate-modified` files but `blast_radius.for_candidate_modified_set` is empty or omits them. **For L1, blast-radius may be empty (L1 work is typically too small to have meaningful blast radius) — the cross-check should accommodate this without halting, but the absent-vs-empty distinction still matters.**

3. **Classification self-check at Step 2 is load-bearing.** Session 5 added an explicit halt at Step 2 for L3 triggers appearing in `rules_fired` despite `computed_level: 2`. **At L1, the analog is broader: ANY L2 or L3 trigger in `rules_fired` while `computed_level: 1` indicates classification disagreement and must halt.** L1's classification is the most strict — it requires ALL of: no external systems, no migrations, no security surface above 3 hits, no trust boundaries, ≤5 new contracts, ≤8 expected cards. Any of those flipping bumps the level.

4. **Halt-condition completeness is four-component: trigger, card note content, activity log content, post-halt disk/artifact state cleanup attestation.** Every halt must specify all four. Reviewers in rounds 3, 4, and 5 surfaced halts missing the post-halt state attestation. **For L1, every halt's post-halt-state line should say something like "No architecture document has been written to disk and no artifact has been submitted at this halt point" (for early halts) or "The architecture document exists on disk at `docs/arch/<file>.md`; no artifact has been submitted" (for halts after the single Write call).**

5. **Plant-watering on orchestrator-describing clauses.** Round 4 surfaced multiple clauses in the boundary contract and Step 11 like "the orchestrator parses the Card Slices section after user approval and creates one workspace card per slice" — these describe orchestrator behavior, not subagent instructions. The fix was to rewrite as direct prohibitions: "Do not create per-slice workspace cards." Same pattern applies anywhere a clause's verb has a non-subagent subject (orchestrator, research agent, auditor, user, plan author, maintainer).

6. **No L2/L3 cross-references.** Round 1 surfaced multiple "(replaces L3's collaborativereasoning MCP)" / "(replaces L3's standalone Phase 10a)" parentheticals that explained design choices in terms of what L3 does. The L1 subagent doesn't need to know what L2 or L3 does — it needs to know what L1 does. Strip all "(replaces ...)" / "(unlike ...)" / "(L2 has X but L1 doesn't)" clauses from the profile body.

7. **Three-pass-write reframing example.** When a reviewer finding pushes you toward a structural change that conflicts with a locked decision, reframe instead of restructure. Session 5 reverted "three-pass" → "two-pass with gate-completion Edits" rather than adding a third pass to the contract. **For L1, watch for similar pressure: a reviewer may suggest adding a placeholder-and-Edit pattern to L1 to "match L2/L3 consistency" — refuse it. L1's single-pass write is a locked decision; the way to enable downstream auditability at L1 is precision of the single Write, not additional passes.**

8. **Section order vs body structure.** Plan §6.4 lists 4 body structure items but the actual compose-l2 profile has 9 sections (matching the L3 pattern). The Preamble's "Execution order" paragraph handles the section-position vs. execution-order mismatch. **Plan §6.5 is similarly terse (4 body items, 10 process steps). Session 6 should expect to use the same 9-section structure with an Execution order preamble, and may amend plan §6.5 in the same commit to enumerate the sections — same pattern as Session 4 amended plan §6.3.**

9. **Fresh-review convergence trajectory.** Rounds 1–5 produced ~30, ~20, ~18, ~21, ~25 findings respectively. The pattern is diminishing-but-bouncing, not strictly monotonic — different sections get attention each round. **Plan for 4–6 rounds; commit at the user's stopping criterion rather than chasing zero findings. Apply all findings each round per locked decision #12 (no grading).**

10. **The `(pre) Read the committed document` pattern is L2/L3-specific.** Session 5 added an explicit `Read` tool call at the start of Step 9 so slice derivation reads from the committed Step 8 document. **L1 is single-pass, so there's no separate "read the committed document" step — the slicing happens in the same pass as the body write. But the equivalent discipline at L1 is: the slicing decision is anchored in the spec's R#/Q# and the bundle's `files_relevant`, not in working memory.**

11. **Verification scope derivation algorithm.** Session 5's Step 9 includes a deterministic case-insensitive lexicographic tiebreak for mixed `contributes to` scopes naming different verification cards. **At L1, Verification scope is typically `local-only` for every slice (independent cards, no shared verification), so the derivation is simpler — but the same three allowed values apply and the field is still required.**

12. **`overlap justified:` substring for R-DOC-7.** Session 5 prescribed the exact phrase `overlap justified: <reason>` in slice Descriptions when allowed-touch lists overlap, because plan §7 R-DOC-7 tests for the substring. **For L1, allowed-touch overlap is non-trivial work to justify (cards should be independent at L1) — but if it happens, use the same `overlap justified: <reason>` phrasing. Mention in Step 9's trap audit that overlap at L1 is unusual enough to warrant the document's Limitations section noting it.**

13. **The ISO/IEC 25010:2023 vocabulary uses nine specific top-level characteristics.** Functional Suitability, Performance Efficiency, Compatibility, Interaction Capability, Reliability, Security, Maintainability, Flexibility, Safety. **At L1, the per-decision verification approach discipline does not apply (no D# decisions), so ISO 25010 vocabulary is not used in the document body the way L2 and L3 use it. ISO 25010 may not be in the Standards table at all unless the spec inherited it — in which case it appears as inherited-from-spec.**

## Design decisions locked in Sessions 2–5 — Session 6 inherits these

1. **No codebase discovery in compose.** Frontmatter excludes every RAG and codegraph tool. Discovery results come from the bundle's design fields. This is the load-bearing rework change.

2. **Compose retains Context7 (`resolve-library-id` AND `query-docs`).** External doc lookup is reasoning-shaped and stays in compose. (L1 may have less use for Context7 in practice since L1 cannot introduce external libraries, but the tools remain available; mid-process discovery of an external library is a classification escalation halt.)

3. **Universal expert-standards activation as Step 1, verbatim.** Non-negotiable.

4. **Freestanding `## Halt condition` section.** Pattern from Sessions 4 and 5: section appears between Output contract and Process in document order; the preamble's "Execution order" list specifies execution as Step 1 first (cross-cutting), then Halt condition (parses only the level field), then Step 2 onward (full design-field ingestion). Halt checks both `verified_level` (orchestrator parameter) AND `bundle.rule_evaluation.computed_level` (bundle field), halts on disagreement with both values surfaced for orchestrator reconciliation.

5. **Bundle alias.** `bundle` is bound in the preamble as an alias for `arch_facts_bundle`; all `bundle.<field>` references are paths into that object. Single binding statement in the preamble, used consistently throughout.

6. **Corrected-bundle semantics.** When the auditor found discrepancies, the orchestrator passes the corrected bundle, and `bundle.rule_evaluation.computed_level` is the auditor's recomputed level (substituted in by the auditor's corrected-bundle construction per `agents/architecture-classification-auditor.md`). When no discrepancies, it is the research agent's original level. In either case it equals `verified_level`. Plan §3 was annotated in Session 4 to make this explicit.

7. **Level representation split.** Machine contracts use numeric 1/2/3. Document marker uses `**Level:** L1`/`L2`/`L3` as a standalone line matching the validation hook's R-DOC-1 pattern `^\*\*Level:\*\* L[123]$` exactly — no leading or trailing content on the same line. Compose translates numeric → marker form.

8. **Element 5 premise sources for V2 (L2/L3 only).** Spec citation, referenced-document citation, bundle-field citation, Context7, Read of files_relevant file, Grep against spec/named document, test reproduction, or explicit "no factual premises — pure design choice". Do not cite a file the bundle did not name in `files_relevant`. Codebase-RAG and CodeGraph removed entirely. **L1 note:** at L1 there are no D# decisions and therefore no element 5 — slices trace directly to R#/Q#; the element 5 discipline carries forward only for compose profiles that produce D# decisions (L2 and L3).

9. **R-DOC-2 is a subsequence check.** Plan §7 was clarified in Session 4: required headings must appear in the specified relative order; optional sections may interleave without failing the rule. L1 sequence per plan §7: `# Architecture —`, `## Goal`, `## Scope`, italic-attestation line, `## Card Slices`, `## Limitations`, `## Standards governing this architecture`, `## Status`.

10. **Two-condition family criterion for inheritance.** `## Inheritance from existing precedents` section is required only when both (a) the architecture addresses a structurally identical problem within the same system as a prior architecture AND (b) it uses the same architectural pattern. Rare at L1.

11. **Reviewer discipline: review the work, not the fixes.** Every follow-up reviewer dispatch is briefed as an independent fresh review of the current file against contract / plan / standards, with no mention of prior findings in the brief. Verification-style briefing anchors the reviewer and is forbidden. Stop dispatching new passes when a fresh pass returns zero findings — or earlier per user choice (Session 5 committed at post-5th-pass state per user's explicit AskUserQuestion answer). Saved as `feedback_review_not_verify.md` in local memory.

12. **All findings apply in full — no grading.** Never label issues as "worth flagging" / "near-miss" / "minor — won't fix." If a reviewer notices it, apply it. This applies recursively to the author's self-review too. Saved as `feedback_no_grading_findings.md` in local memory.

13. **Plant-watering rule.** Every sentence in the profile is an instruction to the subagent the profile defines. No content directed at the orchestrator, user, plan author, or maintainer. Includes meta-content about what other levels (L2, L3) do — the L1 subagent does not need to know what L2 or L3 do; it needs to know what L1 does.

14. **Plan amendments are in scope for the producing session.** When a reviewer finding surfaces a plan-side defect caused by the session's profile work, fix the plan in the same commit. Pattern: Session 3 amended plan §3; Session 4 amended plan §3, §6.3, §7. Session 5 did not require a plan amendment but left a documented note that plan §6.4's body structure list is non-exhaustive (only 4 items while the actual profile structure has 9 sections per the L3 precedent). **Session 6 should consider amending plan §6.5 for the same alignment** — the body-structure enumeration in §6.5 currently lists 4 items but the actual profile will have ~9 sections; the same Execution order preamble pattern from L2 will apply.

15. **Two-pass write at L2/L3, single-pass at L1 (contract §"What stays" item 5).** L2 (Session 5) and L3 (Session 4) write body in pass 1 and slice in pass 2; gate-completion Edits at Step 10 are not additional structural passes. **L1 is single-pass** — the architecture is the slicing, so the document is written in one pass at Step 7 (or whatever number §6.5 assigns to the Write step) with the Card Slices section populated directly.

16. **Halt-condition four-component completeness.** Every halt instruction (in body and in Failure modes) names: (a) trigger condition, (b) card note content, (c) activity log content, (d) post-halt disk/artifact state cleanup attestation. Session 5 added the post-halt state attestation to every halt path after round-3 and round-5 reviewers surfaced gaps.

17. **Failure modes section as standalone reference.** Every halt path in the body has a corresponding Failure modes entry; the Failure modes section is the consolidated reference a future maintainer reads. Both must be consistent on trigger, card note content, activity log content, and post-halt state.

18. **`overlap justified:` substring for R-DOC-7.** Session 5 prescribed the exact phrase `overlap justified: <reason>` in slice Descriptions when allowed-touch lists overlap. The validation hook's R-DOC-7 substring check uses this substring. The colon and reason continuation make the justification human-readable; the substring `overlap justified` (without the colon) is what the hook matches.
